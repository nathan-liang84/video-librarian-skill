# PRD:视频/照片素材智能整理 Skill(代号:VideoLibrarian)

> 版本:v1.2(草案) · 日期:2026-06-03 · 负责人:yfliang84
> 目标形态:一套**跨 Agent 平台通用的 Skill**(OpenClaw / Codex / Claude Code 均可调用)
> v1 范围:**专业剪辑素材库**(场景 A),媒体类型涵盖**视频 + 照片**。通用大众相册整理(场景 B,任意陌生人/生活事件)仍作独立 skill。
> **照片增强已纳入本 skill 规划**:HEIC 格式兼容、垃圾过滤、近重复归组、InsightFace 人脸聚类(剪辑素材视角)—— 详见 [docs/PHOTO_PIPELINE.md](docs/PHOTO_PIPELINE.md)。
>
> 版本演进:v1.1 曾纳入人脸工作流;v1.2 厘清边界——**剪辑素材视角的照片增强(含人脸聚类)纳入本 skill**,通用大众版(任意陌生人、相册/事件整理)保持独立。

---

## 1. 背景与问题

创作者/剪辑团队手里积累了**大量零散视频和照片素材**(空镜、采访、Vlog 片段、B-roll、生活照…),普遍存在:

- 文件名无意义(`IMG_0832.mov`、`DJI_0451.mp4`),靠人脑记不住内容;
- 不知道某个素材里"拍了什么、能不能用、哪一段能用";
- 剪辑时按脚本找素材靠"翻文件夹 + 反复预览",效率极低;
- 没有统一的索引,素材越多越无法复用,等于沉没成本。

**核心诉求**:让 Agent 自动"看懂"每个视频/照片 → 总结 + 打标签 + 规范(且简短)命名 → 汇总成一张可检索的素材总表 → 未来按剪辑脚本快速匹配可用素材。

---

## 2. 目标与非目标

### 2.1 v1 目标
1. 批量"读懂"素材:**视频**走画面 + 语音双通道理解;**照片**走画面理解;产出结构化内容描述。
2. 受控标签体系:保证标签一致,可被精确筛选(而非自由发挥导致同义词泛滥)。
3. 规范且简短命名:按"拍摄时间_人物_场景"规则重命名文件,**安全、可回滚**(人物来自预设名册,见 §4.3)。
4. 一张素材总表:每个素材一行,字段面向"剪辑可用性"设计,含缩略图。
5. 脚本匹配:输入一段剪辑脚本/分镜需求,从总表中召回可用素材 + 推荐时间码。
6. 可断点续跑:处理大量素材时支持中断恢复、增量更新,不重复花钱。

### 2.2 非目标(v1 明确不做)
- 不做通用大众版整理(场景 B)的专属界面/标签——仅预留 schema。
- **v1 不做人脸聚类**:主角固定,改用"预设人物名册 + 参考图引导识别"(见 §4.3),无需聚类模块。
  - 注:**剪辑素材视角**的人脸聚类(把陌生同框人物归簇、再让用户给簇命名)属于**照片增强 P2,在本 skill 实现**(见 [docs/PHOTO_PIPELINE.md](docs/PHOTO_PIPELINE.md) §Phase 2);**仍独立**的只有通用大众相册(任意陌生人长期相册、生活事件/地点/时间线整理,见 §11)。
- 不做视频剪辑/导出本身——只产出"用什么素材、用哪一段"的决策数据。
- 不做实时/在线处理——以批处理离线整理为主。
- 不自建前端 UI——检索体验依托飞书多维表格自带视图或导出的 Excel。

## 3. 目标用户与典型场景

| 用户 | 场景 | 期望 |
|------|------|------|
| 短视频/Vlog 创作者(个人) | 几百个手机/相机素材待整理 | 一键整理 + 之后按需求找片段 |
| 剪辑师 | 接到脚本,要从素材库找匹配镜头 | 输入脚本 → 输出"可用素材清单 + 时间码" |
| 工作室素材管理 | 维护团队公共素材库 | 标签统一、命名统一、可增量入库 |

**核心用户旅程(剪辑场景):**
1. 把视频/照片丢进一个目录(本地 / 云盘挂载目录)。
2. 运行 skill → 自动盘点、理解、打标签、命名、入库。
3. 拿到素材总表(飞书多维表格或 Excel)。
4. 剪辑时:把脚本/分镜喂给 skill → 返回每个分镜对应的候选素材 + 推荐时间码。

## 4. 整体架构

### 4.1 处理管线(Pipeline)

```
┌─────────────────────────────────────────────────────────────────┐

└─────────────────────────────────────────────────────────────────┘
        │
        ▼
┌──────────────┐   ┌───────────────────────────┐   ┌──────────────┐
│ 1. 盘点 Scan │ → │ 2. 抽取 Extract            │ → │ 3. 理解 Understand│
│ 遍历目录      │   │ 视频:关键帧+音轨→ASR       │   │ 多模态看帧/图  │
│ 指纹去重      │   │ 照片:直接读图              │   │ + 读字幕       │
│ 元数据(EXIF/  │   │ 生成缩略图/雪碧图          │   │ → 结构化输出   │
│  ffprobe)     │   │                           │   │              │
└──────────────┘   └───────────────────────────┘   └──────────────┘
                                                          │
        ┌─────────────────────────────────────────────────┘
        ▼
┌──────────────┐   ┌──────────────┐   ┌──────────────────────────┐
│ 4. 打标签 Tag │ → │ 5. 命名 Rename│ → │ 6. 入库 Store(可插拔适配器)│
│ 受控词表+自由 │   │ 时间_场景(_人) │   │ A) 飞书多维表格            │
│ +泛称人物     │   │ 简短+可回滚    │   │ B) JSON旁车+Excel/CSV      │
└──────────────┘   └──────────────┘   └──────────────────────────┘

                                  ┌──────────────────────────┐

```

> 人脸聚类作为**人物增强挂载点**:管线在第 2 步与第 4 步之间预留挂载点。**照片增强 P2(本 skill)**在此接入人脸检测/聚类/簇命名(剪辑素材视角,见 docs/PHOTO_PIPELINE.md §Phase 2);通用大众相册整理仍独立,复用同一挂载点。两者都不改动其余环节。

### 4.2 内容理解:画面 + 语音双通道(视频)/ 画面(照片)

- **画面通道**
  - 元数据:视频用 `ffprobe`(时长/分辨率/帧率/编码/码率/大小/创建时间/GPS);照片读 **EXIF**(拍摄时间/设备/GPS/参数)。
  - 视频抽关键帧:**场景切换检测**(`select='gt(scene,0.4)'`)+ **每 5 秒均匀采样兜底**;每分钟 ≤6 帧、单视频 ≤36 帧;短视频(<10s)≥3 帧。
  - 照片:直接作为单图输入。
  - 关键帧/图交多模态模型 → 场景、景别、运镜、主体(泛称)、动作、情绪、色调/光线、画质评估。
- **语音通道(仅视频)**
  - `ffmpeg` 抽音轨 → ASR 转写(优先本地 `faster-whisper`;无则降级云端 ASR;无音轨跳过)。
- **融合**:画面理解 + 语音转写 + 文件元数据 + 人物名册 → 模型产最终总结、标签、可用片段判定,输出统一 JSON。

### 4.3 人物名册 + 参考图引导识别(v1 方案,主角固定)

v1 以**某个固定主角(由用户在配置时指定)**为核心剪辑对象,人物高度集中,v1 不启用人脸聚类(聚类作为照片增强 P2 在本 skill 另行落地,见 §11.1 与 docs/PHOTO_PIPELINE.md)。v1 改用轻量方案:

1. **预设人物名册(config 配置)**:登记本库人物及关系,例如
   - 主角:`<用户指定的主角名>`(附 1–2 张参考图路径)
   - 关系人/宠物:`男朋友`、`闺蜜`、`宠物狗`…(可附参考图,选填)
2. **参考图引导识别**:理解每个片段时,把主角参考图 + 关键帧一起喂给 M3,让它**只从名册里判断**画面里有谁,输出组合标签,如:
   `主角`(单人)/ `主角和男朋友` / `主角和宠物狗` / `多人` / `空镜(无人)`。
3. **直接命名 + 入表**:识别结果即 `subjects` 字段,并参与文件命名(见 §5.3)。
4. **名册可扩展**:新增固定人物时往 config 加一条 + 参考图即可,不改管线。

> 与人脸聚类的区别:本方案针对**已知少量人物**,靠多模态模型看图直接归类,不建人脸向量库。需要对**陌生同框人物**建簇时,走照片增强 P2 的人脸聚类(本 skill);面向**任意陌生人长期相册**的完整聚类 + 逐一命名仍属大众版(独立,见 §11)。

### 4.4 数据层:可插拔双模式(已选方案)

核心管线产出**统一中间产物**(每个素材一份标准化 JSON 记录),由"数据层适配器"决定落到哪里。管线不变,只切换输出端。

- **模式一 · 飞书多维表格适配器**(优先)
  - 通过飞书开放接口写入"素材库表";字段映射见 §5;缩略图作为附件/封面。
  - 复用多维表格自带的**筛选、分组视图、附件预览**做检索。
- **模式二 · 云盘 + 旁车适配器**(用户只把素材存百度网盘等云盘、无数据库)
  - 给**每个素材生成同名 JSON 旁车文件**(`20240715_海边_全景_01.mp4` → 同名 `.json`),随素材一起存放/同步;缩略图存同目录 `.thumbs/`。
  - 再**汇总成一张 Excel/CSV 总表**(`_素材总表.xlsx`),供整体浏览与导出。
  - 旁车意义:素材搬家/换设备时元数据不丢,可随时从旁车重建总表。

> 适配器自动判断(是否配置飞书 / 是否仅云盘),也可在配置里显式 `store: feishu | sidecar`,两种可同时双写。

### 4.5 跨平台 Skill 结构

为兼容 OpenClaw / Codex / Claude Code,采用"**说明文档 + 独立脚本**"形态,逻辑落在可移植的 Python 脚本里,Agent 负责调度与模型推理。

```
video-librarian-skill/
├── SKILL.md               # 技能说明 + 触发条件 + 调用流程(Agent 读这个)
├── PRD.md                 # 本文档
├── config/
│   ├── config.example.yaml   # 数据层模式、命名规则、抽帧策略、模型、人物名册(主角+参考图)
│   └── vocab.yaml            # 受控标签词表(景别/运镜/情绪/场景/用途…)
├── scripts/
│   ├── 00_detect_env.py      # 环境探测(ffmpeg/ASR/数据层)
│   ├── 01_scan.py            # 盘点 + 指纹去重 + EXIF/ffprobe
│   ├── 02_extract.py         # 抽关键帧 + 抽音轨/ASR + 缩略图/雪碧图
│   ├── 03_understand.py      # 调多模态 + 融合 → 结构化 JSON
│   ├── 04_tag_name.py        # 受控打标签 + 生成新文件名(dry-run/apply)
│   ├── 05_store.py           # 数据层适配器(feishu / sidecar 双模式)
│   └── 06_match.py           # 脚本匹配检索
├── adapters/
│   ├── store_feishu.py       # 飞书多维表格
│   └── store_sidecar.py      # JSON 旁车 + Excel/CSV
└── state/
    ├── manifest.json         # 处理进度/指纹/版本(断点续跑)
    └── rename_log.json       # 改名映射(可回滚)
```

> **可移植性原则**:脚本只依赖命令行工具(ffmpeg/ffprobe)与少量 Python 库;模型与 ASR 调用走抽象层,便于切换本地/云端;缺依赖时给明确指引而非静默失败。
> **扩展**:人脸模块作为可选脚本 `02b_faces.py` + `state/faces.db` 预留位,**照片增强 P2(本 skill)**启用,大众版复用同一位。

## 5. 数据 Schema(v1,面向剪辑可用性)

> 每个素材 = 一条记录。`media_type` 区分视频/照片;人物字段 v1 只填泛称,可手动补名,未来由人脸模块自动填充。

### 5.1 字段定义

| 分组 | 字段 | 类型 | 说明 |
|------|------|------|------|
| 识别 | `id` | string | 文件内容指纹(SHA1 前 16 位),去重主键 |
| 识别 | `media_type` | enum | video / photo |
| 识别 | `content_kind` | enum | **P1b-1 新增**:目录级内容类型 `video` / `photo` / `mixed`,01_scan 聚合目录下所有媒体类型后写入。仅视频 → video、仅照片 → photo、两者都有 → mixed;旧数据 null 时消费者回退到 `media_type` |
| 识别 | `original_name` | string | 原始文件名 |
| 识别 | `new_name` | string | 规范化新文件名(简短) |
| 识别 | `path` | string | 当前路径 / 云盘相对路径 |
| 识别 | `thumbnail` | string | 缩略图路径 / 附件(视频含雪碧图) |
| 技术 | `duration_sec` | number | 时长(视频) |
| 技术 | `resolution` | string | 如 `3840x2160` |
| 技术 | `fps` | number | 帧率(视频) |
| 技术 | `codec` | string | 编码(视频) |
| 技术 | `filesize_mb` | number | 文件大小 |
| 技术 | `shot_at` | datetime | 拍摄时间(EXIF/元数据,缺失则文件时间) |
| 技术 | `gps` | string | 经纬度(若有) |
| 技术 | `device` | string | 拍摄设备(若有) |
| 内容 | `summary` | string | 一句话总结 |
| 内容 | `description` | text | 详细内容描述 |
| 内容 | `scene` | enum[] | 场景(受控词表) |
| 内容 | `subjects` | string[] | **人物(来自预设名册,如 主角 / 主角和男朋友 / 主角和宠物狗 / 多人 / 空镜),由 M3 参考图引导识别;名册外的人记 `多人`** |
| 内容 | `actions` | string[] | 主要动作/事件 |
| 剪辑 | `shot_type` | enum | 景别:远景/全景/中景/近景/特写(受控) |
| 剪辑 | `camera_move` | enum[] | 运镜(受控,视频) |
| 剪辑 | `mood` | enum[] | 情绪/调性(受控) |
| 剪辑 | `lighting` | enum | 光线(受控) |
| 剪辑 | `quality_score` | number | 画质评分 1–5 |
| 剪辑 | `has_speech` | bool | 是否有对白/人声(视频) |
| 剪辑 | `transcript` | text | 语音转写全文(视频) |
| 剪辑 | `usable_clips` | json[] | 可用片段:`[{start, end, reason}]` 时间码(视频) |
| 剪辑 | `suggested_use` | enum[] | 建议用途(受控) |
| 剪辑 | `tags` | string[] | 自由标签(受控词表外补充) |
| 管理 | `confidence` | number | 模型理解置信度 |
| 管理 | `status` | enum | pending/done/needs_review/failed |
| 管理 | `schema_version` | string | schema 版本 |
| 管理 | `processed_at` | datetime | 处理时间 |

### 5.2 受控词表(vocab.yaml,节选)

为保证"可筛选",以下字段取值受控,模型必须从枚举里选(可多选);确实没有才进 `tags`。

- `shot_type`:远景 / 大全景 / 全景 / 中景 / 近景 / 特写 / 大特写
- `camera_move`:固定镜头 / 推 / 拉 / 摇 / 移 / 跟随 / 升降 / 航拍 / 延时
- `mood`:欢快 / 平静 / 温暖 / 紧张 / 悬疑 / 壮阔 / 忧伤 / 治愈 / 高能
- `scene`:城市 / 自然风光 / 室内 / 街道 / 海边 / 山地 / 美食 / 人物特写 / 交通 / 夜景 / 活动现场
- `suggested_use`:片头 / 片尾 / 转场 / B-roll / 空镜 / 采访同期 / 氛围铺垫 / 高潮段落
- `content_kind`(P1b-1):`video` / `photo` / `mixed` —— 目录级内容类型,由 01_scan 聚合推断;消费者可用 `Record.effective_content_kind` 获取(None 时回退 `media_type`)

> 词表可由用户自定义扩展,集中在 vocab.yaml 维护。

### 5.3 命名规则(简短,可配置)

**默认模板(短优先,时间_人物_场景):**

- 照片:`{日期}_{人物}_{场景}_{序号}` → `20240715_主角_海边_01.jpg`
- 视频:`{日期}_{人物}_{场景}_{景别}_{序号}` → `20240715_主角_海边_全景_01.mp4`
- 多人物组合:`20240715_主角和男友_咖啡馆_中景_01.mp4`、`20240715_主角和狗狗_公园_全景_01.mp4`
- 无人物(空镜)时人物段省略:`20240715_城市_全景_02.mp4`。

规则约束:
- 字段缺失自动省略对应段,保持短;
- 非法字符过滤、长度上限、跨平台兼容(避免 `:/\?*` 等);中文/英文模板可切换;
- **改名安全**:先 dry-run 输出"旧名→新名"清单供确认;apply 时写 `state/rename_log.json`,**支持一键回滚**;同名冲突自动加序号;**默认不删原文件**。

> v1 命名里的"主体"段已是名册里的具体人名(由用户指定);照片增强 P2 接入人脸聚类后,簇命名结果同样回填该段,可自动识别更多人物,规则不变。

## 6. 检索 / 脚本匹配(v1 关键能力)

输入:一段剪辑脚本 / 分镜表(自然语言,如"开场要一个城市清晨的空镜,暖色调,3 秒以上")。

流程:
1. 解析脚本 → 拆成若干"镜头需求"(场景、情绪、景别、时长、关键词)。
2. **结构化过滤 + 语义排序**:先用受控字段硬过滤,再用 summary/description 语义相似度排序。
3. 输出每个需求的**候选素材清单**:文件名 + 推荐 `usable_clips` 时间码 + 匹配理由 + 置信度 + 缩略图。

输出示例:
```
镜头1 城市清晨空镜/暖调/≥3s
  ① 20240715_城市_全景_02.mp4  片段 00:04–00:09  暖调日出 ★★★★
  ② 20240320_城市_远景_11.mp4  片段 00:00–00:06  偏冷需调色 ★★★
```

## 7. 多模态模型与引擎选型(已选:MiniMax M3 + M2.7 分工)

**核心策略:M3 只用于"看画面",M2.7 处理所有纯文本步骤,ASR 用本地。** 各用各擅长的,把贵的多模态调用降到最少。

| 用途 | 选型 | 价格(官方,人民币/百万 token) | 说明 |
|------|------|------|------|
| **画面理解** | **MiniMax M3**(原生多模态,支持图片+视频) | 输入 5.6 / 输出 8.4(≤512K,近期 7 天五折);缓存读 0.42 | 看关键帧/照片 → 场景/景别/情绪/画质等结构化内容。必须用多模态的地方 |
| **纯文本步骤** | **MiniMax M2.7**(纯文本,**不支持图像**) | 输入≈$0.30 / 输出≈$1.20 | ① 融合 ASR+画面描述做总结;② 受控词表打标签;③ 脚本解析与匹配 |
| **语音 ASR** | faster-whisper(本地) | 免费 | 本地优先省钱保隐私;无则降级云端 ASR |
| 备选(可换) | 豆包 Doubao Vision / Qwen-VL | — | 模型走抽象层,需更低单价时对比替换"看画面"环节 |

> **重要纠正**:M2.7 **不支持图像输入**,不能用来看帧/照片;能看画面的是 M3。两者分工而非二选一。

**省钱四招:**
1. **抽帧喂 M3,不整段丢视频**——M3 虽支持原生视频,长视频按 token 算贵;抽关键帧(≤36 帧/视频)控成本,极短片段才整段喂。
2. **两档处理**——先便宜跑"快扫"建索引,只对要精修的素材再精跑;按指纹去重不重复调用。
3. **吃 7 天五折**——首批大批量入库赶在五折期跑完。
4. **缓存固定提示词**(指令/vocab,0.42 元/M);并核实 MiniMax **Token Plan 订阅套餐**(¥49/6亿token 起,称支持全模态)是否适用于 API 批处理——若适用,比按量便宜一个数量级。

## 8. 非功能需求

- **可断点续跑 / 增量**:以指纹为键记录进度;重复运行只处理新增/变更;失败项标 `failed` 可单独重试。
- **成本可控**:抽帧上限;两档处理;同一文件不重复调模型。
- **大批量**:并发处理(可配并发度);进度可见;支持分批/分目录。
- **安全**:改名可回滚;默认不删原文件;旁车不污染原素材。
- **隐私**:本地优先(ffmpeg/本地 ASR);上云步骤需用户显式同意。
- **可移植**:同一套脚本在 OpenClaw/Codex/Claude Code 行为一致;缺依赖给明确指引。

## 10. 风险与对策

| 风险 | 对策 |
|------|------|
| 大批量成本/耗时高 | 抽帧上限 + 两档处理 + 指纹去重 |
| 标签不一致致检索失效 | 受控词表强约束,自由标签兜底 |
| 改名误操作不可逆 | dry-run 确认 + rename_log + 一键回滚 + 不删原文件 |
| MiniMax 是否支持图像未知 | 选型阶段先核实,默认主力用 Qwen-VL |
| 平台 ffmpeg/模型能力差异 | 环境探测 + 降级 + 明确报错 |
| 模型理解不准 | 输出 confidence,<0.6 标 needs_review |
| 云盘搬家元数据丢失 | 旁车 JSON 随文件走,可重建总表 |

## 11. 扩展方向

### 11.1 照片增强(已纳入本 skill,分阶段实现)

> 详见 [docs/PHOTO_PIPELINE.md](docs/PHOTO_PIPELINE.md)。

剪辑素材视角的照片专项能力,分两阶段并入本仓库:

- **Phase 1(轻量,立即见效)**:HEIC 格式兼容 + EXIF 旋正 + Live Photo 配对 + 垃圾过滤(截图/翻拍/表情包)+ 近重复/连拍归组(pHash)。  
  关键设计(分支终态,详见 [docs/PHOTO_PIPELINE.md](docs/PHOTO_PIPELINE.md) §4.1):
  - 垃圾记录 `status=junk`——跳过 02/03/04 不烧钱,以最小 record 入库(`is_junk=true` 持久标记);06 不召回;`--include-junk` 误判时重跑。
  - 近重复/连拍**非代表成员** `status=grouped`——代表正常精理解,成员跳过 02/03/04、以最小 record 入库(`is_representative=false`+`group_id`);06 不直接召回,经 `group_id` 可从代表展开发现。
- **Phase 2(可选,需安装本地模型)**:InsightFace(ArcFace) 本地人脸聚类 + M3 贴标签 → 回填 `subjects` 与名册;人脸数据只存本地。  
  挂载点:管线 `01b` 之后新增 `face_cluster` pass(`state/faces.db` 预留);Phase 1 合并后可独立安装。

### 11.2 通用大众版(场景 B,仍作独立 skill)

面向普通人整理生活照片/视频;核心差异在于**任意陌生人建库**、按生活事件/地点/时间线组织——超出剪辑素材视角,不与本仓库混合。

- 完整规划见 [photo-librarian-skill](https://github.com/nathan-liang84/photo-librarian-skill)(人脸归类/地理/事件/相册/自然语言检索)。

## 12. 验收标准(v1)

1. 给定含 ≥50 个视频+照片的目录,完成"盘点→抽取→理解→打标签→命名→入库"全流程,可断点续跑。
2. 总表每行字段完整,受控字段取值均在词表内,含缩略图。
3. 改名支持 dry-run 预览与回滚,无数据损坏;命名简短符合"时间_场景(_人)"。
4. 飞书模式与旁车模式任一可用;旁车能重建总表。
5. 输入含 ≥3 个镜头需求的脚本,返回每个需求的候选素材 + 时间码,人工抽查命中合理。
