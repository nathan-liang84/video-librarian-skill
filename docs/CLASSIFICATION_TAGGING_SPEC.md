# 照片 / 视频分类与标签**一致性**规范 v1.0(approved)

> **本文目的**:解决当前管线"识别内容随意、打标签随意——基本相同的照片/视频,跑出来的分类和标签差很大"的问题。
> **硬目标**:**同一张照片/同一段视频,每次读出来必须落入相同的分类与受控标签。**
> **本文状态**:**approved(2026-06-11)**。外部评审(Fable 5)批准进入实施,4 处修订已回写(见 §2.1/H2、§4/L1、§6/M1、§5.3);§9 由"开放问题"改为"评审结论"。**冻结后变更走 ADR + bump 版本。**

---

## 1. 背景

本项目是一个跨 Agent 平台的视频/照片**素材**整理工具:批量"读懂"素材 → 打标签、规范命名 → 汇总成可检索的素材总表 → 按剪辑脚本匹配可用片段。理解阶段用一个**用户自配的多模态模型**看画面、一个文本模型融合语音转写做总结打标签。

当前问题:在大批量识别归档后,发现**同一张(或基本相同的)照片/视频,两次跑出来的分类、标签、内容描述相差很大**,导致总表无法用于稳定检索——这正是素材库的核心价值所在。

---

## 2. 问题定性:这是**两个叠加**的病,不是一个

| 病 | 表现 | 根因 |
|----|------|------|
| **A. 不确定性(non-determinism)** | 同一输入两次跑,结果不同 | 模型自由采样 + 缺内容寻址缓存 + 视频送帧不一致 + 自由文本字段 + 校验时机漂移 |
| **B. 无规范分类体系** | 即使在"允许值"里也乱选 | 受控词表是**扁平枚举**,没有闭集、互斥的"主分类",字段间无消歧规则 |

**关键认识:只治 B(做一张漂亮的分类表)是不够的。** 只要 A 还在,同一张图在同一套词表里照样飘。**必须先用工程手段锁住确定性(治 A),再叠加规范体系(治 B),最后用评测证明真的治住了。**

### 2.1 代码层面的漂移源(诊断实证)

对现有管线逐文件审查后,定位到 21 处可导致同一输入产生不同输出的漂移源,其中 4 处为高危(下表)。`file:line` 为现有代码位置,供评审者核对诊断是否准确。

| # | 漂移源 | 证据 | 危害 |
|---|--------|------|------|
| H1 | **采样不确定** | `lib/models.py:113` `temperature=0.2` 写死;**无 seed / top_p / 结构化输出**;`_extract_json` 仅正则解析、不对 schema 校验 | 同输入每次采样不同 → 字段值漂 |
| H2 | **缺内容寻址结果缓存** | 状态机已防"正常流程内同一条被重复处理",但**无内容指纹缓存**:`03_understand.py:125-131` 调模型前不做内容指纹比对,`lib/models.py:121-136` 仅网络错误重试。一旦 `--rebuild`、跨记录撞同内容、或重跑历史,就会**对同一内容重新问模型**,结果不复用 | 同内容跨次/跨记录不收敛 |
| H3 | **视频送帧不一致** | `03_understand.py:102-103` quick 档送 4 帧、refine 档送 36 帧;同一视频先 quick 后 refine,**模型看到的帧集完全不同** | 同一视频"内容"必然不同(视频头号病因) |
| H4 | **校验用 live config 非快照** | `validate.py:49-53` 人物名册在校验时从当前 config 现取;`migrate_vocab.py` 词表重映射无版本、非幂等 | 中途改 config/词表 → 同一旧记录校验/结果就变 |

另有 17 处中低危,典型:`subjects`/`tags`/`summary` 等自由文本字段无落地枚举约束;文本模型会**静默覆盖** `main_subject`/`subject_kind`;`record.schema.json` 不在 schema 层强制枚举(只在 Python 代码里校验);多选字段顺序虽 `sorted()` 但语义优先级丢失;人物匹配完全交给模型判断、无确定性串/向量匹配兜底。

## 3. 一致性契约:先讲清"保证到什么程度"

不同字段的**保证级别不同**,这点必须先约定,否则无法验收:

- **🔒 受控字段(分类 + 受控标签):硬保证可复现。**
  `primary_category`、各受控 facet(scene / activity / shot_type / camera_move / mood / lighting / suggested_use / quality)、`subjects`、`subject_kind`。
  **目标:同一指纹输入 → 逐字段完全一致。**
- **〜 自由散文(summary / description / keyword):尽力稳定,不强求逐字一致。**
  这些字段天生主观;锁死采样 + 缓存能让它"基本稳定",但承诺"逐字相同"既不现实也无必要——**检索与归档依赖的是受控字段,不是散文。**

> 一句话:**我们保证的是"分类与受控标签可复现",不是"模型每个字都一样"。** 用户诉求("落入相同的分类和标签")正好落在硬保证这一侧。

## 4. 确定性引擎(治 A):7 层防御,叠加使用

不是七选一,是分层兜底——从"把模型调用变成纯函数"到"近重复直接继承",逐层把熵挤掉。

| 层 | 名称 | 做什么 | 主要消除 |
|----|------|--------|----------|
| **L1** | 固定采样 | `temperature=0` + 固定 `seed` + 固定 `top_p`;prompt 模板、`vocab_version`、`model_id` 全部钉死。模型调用 ≈ 纯函数 | H1 |
| **L2** | 固定输入 | 抽帧与省钱档位**解耦**:`02_extract` 永远产出同一套**规范帧**并存其哈希;quick/refine 只在"送几帧"上做确定性子采样(同 cap → 同帧),**绝不让 refine 看到 quick 没看过的帧**。照片输入即字节本身 | H3 |
| **L3** | 结构化输出(源头约束) | 用 JSON-schema / function-calling **强制枚举**,模型只能吐允许的分类值;解析后**对 schema 校验**,违例即重试。把自由文本字段压到最小 | H1、自由字段 |
| **L4** | 归一化吸附(事后兜底) | 确定性规范化层:残留自由文本经"同义词词典 +(可选)固定 embedding 最近邻 + 确定性 tie-break"吸附到 canonical 值;多选字段按 canonical 顺序**排序** | 自由字段、顺序漂 |
| **L5** | 指纹缓存(终极保证) | `content_fingerprint = hash(规范帧 + 转写 + prompt_version + vocab_version + model_id + 采样参数)`;**指纹命中 → 直接返回存档记录,根本不再调模型** | H2 |
| **L6** | 近重复继承 | 理解**之前**先 pHash 归组;**只理解组代表一次,成员继承代表的分类/标签**,不各自重推 | "基本相同的图"漂 |
| **L7** | 快照绑定 | 理解时把 `vocab_version + roster_snapshot + thresholds_version` 写进记录;后续校验用快照而非 live config | H4 |

> **L1 的诚实声明(回写修订 2)**:`temperature=0` + `seed` **并非比特级确定**——多数 provider 不保证可复现采样,硬件/批处理/版本差异都会让同参数同输入产生不同 token。因此**真正的硬保证来自 L3(枚举约束)+ L4(吸附)+ L5(指纹缓存)**,L1/L2 只是把熵降到吸附层能稳定收敛的范围。**若 L4 启用 embedding 最近邻吸附,该 embedding 模型版本必须并入 `content_fingerprint`**(否则换 embedding = 吸附结果变 = 缓存失效未被察觉)。

### 4.1 三句话理解

- **同一张图** → 指纹命中,直接复用(L5),根本不再问模型。
- **基本相同的图** → 归组,成员继承代表的标签(L6),而不是各跑各的模型。
- **全新的图** → 锁采样(L1)+ 定帧(L2)+ 强枚举(L3)+ 吸附(L4)保证落点稳定;config 不许中途捣乱(L7)。

> L5 + L6 是直接面向"同一/基本相同照片落入相同分类标签"诉求的两层;L1–L4 保证"第一次见到时"的落点本身就稳;L7 防止外部状态把已稳的结果搅乱。

## 5. 分类体系规范(治 B):闭集、分层、分面

核心改造:从**扁平枚举** → **单选主分类 + 正交分面**,且每个字段配**书面决策规则**。
**规则才是一致性的真正来源,枚举本身不是**——两个标注者(或两次模型调用)能否落到同一格,取决于有没有明确的消歧规则。

### 5.1 分面模型

| 维度 | 基数 | 闭/开 | 说明 |
|------|------|-------|------|
| `primary_category` | **单选** | 闭集 | 互斥主分类,由决策树定优先级 |
| `scene`(地点) | 多选 | 闭 + 毕业 | 沿用现有具体地点表(健身房/花店/餐厅/咖啡厅/家里/户外街道/商场/酒店/公园/海边/活动现场/…) |
| `activity`(活动/事件) | 多选 | 闭 + 毕业 | 新增:吃饭 / 健身 / 拍摄 / 通勤 / 购物 / 聚会 / … |
| `subject_kind`(主体类型) | 单选 | 闭集 | 人物 / 物品 / 建筑 / 风景 / 动物 / 食物 / 其他 |
| `shot_type` / `lighting` | 单选 | 闭集 | 沿用现有词表 |
| `camera_move` / `mood` / `suggested_use` | 多选 | 闭集 | 多选字段强制 canonical 排序 |
| `quality` | 单值(1–5) | — | 画质评分 |
| `tags` | 多选 | 开放 | 受控集未覆盖的特征;走"毕业"机制定期归并入受控集 |

> **"闭 + 毕业"** = 默认闭集;新概念先进 `tags` 自由字段,定期评审后**毕业**为受控枚举值并 bump `vocab_version`,既不堵新概念、又不让自由字段长期污染检索。

### 5.2 `primary_category` 闭集(素材域,定版 9 类)

`人物` · `活动事件` · `美食` · `风景空镜` · `建筑空间` · `物品产品` · `宠物动物` · `交通旅途` · `其他`

> 评审定版(Q1):9 类保持,v1 **不**拆"活动事件"(避免过早细分;细分需 ADR + 金标重标)。

### 5.3 决策规则(消歧 = 一致性的真正来源)

写进**模型 prompt** 和**人工标注手册**,二者同源。

**主体优先级铁律(单选 `primary_category` 的 tie-break)**:
- **优先级链:人 > 动物 > 食物/物品 > 建筑/场景。** 多类并存时取最高者为 `primary_category`,其余进对应 facet(`activity` / `scene`)。
- **该链只含"主体级"类别。`活动事件` 是"事件级"类别,不在链内**:**仅当画面无单一主体主导**(群体远景/全景、事件氛围镜头、无明确人脸/主体特写)时,才落 `活动事件`(回写修订 4 / Q2)。有单一主体主导 → 永远落该主体类别,活动进 `activity` facet。

**高频歧义补规则(Q1/Q2 定版)**:
- *餐厅里有人吃饭* → 主体是人(中近景人脸/人物动作)判 `人物`,食物进 `scene`/`activity`;主体是食物特写判 `美食`。
- *地标/景点前自拍或人物入镜* → `人物`(人是主体,地标进 `scene`)。
- *宠物与主人同框* → `人物`(`subjects` 仍保留宠物条目,不丢信息)。
- *车窗 / 机翼外的风景* → `交通旅途`(出行语境优先于纯风景),非 `风景空镜`。
- *拿不准* → 落 `其他` + `needs_review`,**绝不随机挑一个**。

### 5.4 `vocab.yaml` v2 结构要点

- 分面结构 + **每字段显式标 cardinality**(现状只藏在 `validate.py` 的 SINGLE/MULTI 列表里,词表本身没写)。
- 顶部新增 `vocab_version`(语义化版本号)。
- 声明多选字段的 canonical 排序规则。
- 写明"自由 `tags` 如何毕业进受控集"的治理流程,**含自动化**:某 tag 出现频次 > N 自动进毕业候选队列(Q4)。
- `record.schema.json` 升级为**强校验**(`enum` / `oneOf`),不再只当文档——杜绝绕过 Python 直接写非法值。

## 6. 评测与门槛(证明真的稳了——没有这个,一切是空话)

新建 `scripts/eval_consistency.py` + CI 闸。**这是验收"同一张照片 → 同样标签"的尺子。**

| 指标 | 怎么测 | 建议门槛 |
|------|--------|----------|
| **M1 同图复现** | 见下方双模式 | 见下方双模式 |
| **M2 近重复一致** | pHash 同组照片共享 `primary_category` + facet Jaccard | 同组 `primary_category` 一致率 **≥ 0.95** |
| **M3 金标一致** | vs 人工标注金标集,Cohen's κ,**分客观/主观档**(Q5) | **客观字段**(`primary_category`/`shot_type`/`lighting`/`subject_kind`)**κ ≥ 0.8 硬闸**;**主观字段**(`mood`)**κ ≥ 0.55 仅报告不拦截** |
| **M4 覆盖率** | 非 `其他` 落点比例 | **≥ 0.9**(太多"其他"= 体系没覆盖好) |

**M1 双模式(回写修订 3)**——两种模式各自验收,不可混为一谈:
- **M1-a cache-on(契约测试,硬闸 100%)**:开 L5 指纹缓存,同一条跑 K 次必须逐字段精确命中缓存。这测的是"缓存契约不破",**必须 100%**,破了即代码 bug。
- **M1-b bypass-L5(真正的信息量,先 report-only)**:**绕过 L5**,逼真实模型链路跑 K=5 次,量 `primary_category` 精确匹配率 + 受控 facet Jaccard。这才反映 L1–L4 的真实稳定度。**先 report-only 收基线**,基线稳定后再把 `primary_category` 100% / facet Jaccard ≥ 0.95 转硬闸。

- **金标集**:抽 ~150–200 条,覆盖各 `primary_category`,双人标注定 ground truth。
- **A/B**:用同一套金标比较"开了哪几层 L"对 M1–M4 的提升,**数据驱动**决定哪些层必须上、哪些可省。
- **CI 闸**:M1-a 进 CI 硬闸;M1-b 先 report-only,达基线后转硬闸。

## 8. 边界与风险(诚实声明,不承诺做不到的)

- **主观字段(如 `mood`)做不到 100% 确定** → 锁闭集 + 缓存,接受残余;靠 M1-b/M3 主观档量化而非假装为零。
- **模型版本漂移** → `model_id` 进指纹,但用**配置层语义 model id(provider + 模型名),非精确 build id**(Q7):provider 静默小升级时缓存是护盾、不触发全量重算;**显式 `--rebuild` 才吃新模型**。这是特性不是 bug,需写进文档。
- **散文不保证逐字一致**(见 §3),只保证分类 + 受控标签。
- **边界 case 靠书面决策规则**(§5.3)收敛;决策树覆盖不全的组合会落 `其他` + `needs_review`,由人工一键确认。
- **结构化输出依赖 provider 能力**:并非所有 OpenAI 兼容 provider 都支持 function-calling / JSON-schema;不支持时降级到"强 prompt 约束 + L4 吸附"(见 §9-Q3)。

## 9. 评审结论(Fable 5 评审 + Owner 确认,逐条定版)

> 原"开放问题"已拍板。实施以下结论为准;再改走 ADR。

1. **Q1 分类粒度** → **9 类保持,v1 不拆"活动事件"。** 补规则:`交通旅途` vs `风景空镜` 归属——车窗/机翼外风景算 `交通旅途`(§5.3 已写)。
2. **Q2 决策树完备性** → 补铁律:**主体优先级链只含主体级类别;仅当无单一主体主导(群体远景/全景、事件氛围镜头)时落 `活动事件`**。高频歧义补规则:地标前自拍 = `人物`;宠物与主人同框 = `人物`(`subjects` 保留宠物)。均已写进 §5.3。
3. **Q3 结构化输出可移植性** → **三级降级链:`json_schema` → `function-calling` → 强 prompt + L4。** L4 作为**独立达标层**建设,不依赖 L3 能否成功。`00_detect_env` 加能力探测(含推理模型 + json_schema 兼容性测试),按探测结果选档。
4. **Q4 一致性 vs 召回** → **不冲突。** 全文检索靠散文字段,精确筛选靠闭集 facet,两条路。`tags` 毕业机制加**自动化**:出现频次 > N 自动进毕业候选队列(§5.4)。
5. **Q5 门槛取值** → **分档**:客观字段(`primary_category`/`shot_type`/`lighting`/`subject_kind`)κ ≥ 0.8 **硬闸**;主观字段(`mood`)κ ≥ 0.55 **仅报告不拦截**。M1 facet Jaccard 0.95、M4 0.9 取值合理(§6 已落)。
6. **Q6 近重复继承风险** → **继承全部分类 facet,记录 `inherited_from`**;`quality_score` **本地计算不继承**(各帧/各图画质本就不同);**组代表 `subject_confidence` 低于阈值时整组进 review**(防"代表正脸、成员背影"被错误继承)。
7. **Q7 缓存失效边界** → 用**配置层语义 model id(provider + 模型名),非 build id**;provider 静默升级时缓存是护盾,显式 `--rebuild` 才吃新模型(§8 已落)。
