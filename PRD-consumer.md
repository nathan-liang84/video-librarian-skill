# PRD:通用大众版 — 个人照片/视频库智能整理(场景 B)

> 版本:v0.1(规划草案) · 日期:2026-06-03 · 负责人:yfliang84
> 目标形态:与 v1 专业剪辑库(见 [PRD.md](PRD.md))**共用同一套底层管线与记录契约**的消费级能力。
> 关系:v1 先行落地并验证;本版在其之上扩展"人脸归类 / 地点 / 事件 / 相册 / 自然语言检索",面向**普通人**。

---

## 1. 背景与问题

普通人手机/相机/云盘里堆着**成千上万张照片和视频**,典型痛点:

- 文件名是 `IMG_4821.HEIC`,几万张混在一起,想找一张得翻到崩溃;
- 想找"**去年夏天孩子在海边**那几张""**和爸妈一起过年**的视频"——没有任何办法按人物/时间/地点/事件检索;
- 同一个瞬间连拍十几张几乎一样的,占空间又挑不出最好的一张;
- 想给某个人、某次旅行做个相册,只能手动一张张挑;
- 担心隐私:家人照片不愿上传到不可控的云端做人脸分析。

**核心诉求**:让 Agent 自动看懂每张照片/视频 → **识别人脸并由我命名** → 按 时间/人物/地点/场景/事件 打标签归类 → 我能用大白话随时筛选、找回忆、一键成册,而且**人脸等隐私数据不出我的设备**。

---

## 2. 目标与非目标

### 2.1 目标
1. **人脸归类(本版核心)**:本地检测人脸 → 自动聚类 → 用户给每个人命名(可合并/拆分)→ 按人物检索;新照片自动认人。
2. **地点**:读 GPS → 反查地名(国家/城市/地标),按地点筛选。
3. **事件聚类**:把"同一天/同一地点/同一活动"的照片视频自动归成**事件/相册**(如"2024春节·奶奶家""三亚旅行")。
4. **近似去重 + 选最佳**:连拍/重复照片自动分组,挑出最清晰、人脸最好、表情最佳的一张。
5. **生活化标签**:场景(海边/餐厅/公园…)、内容、节日/活动,面向回忆而非剪辑。
6. **简短命名 + 可选归档**:`时间_人物_地点/事件`;可选按"年份/事件"整理到文件夹。
7. **自然语言检索**:"去年夏天孩子在海边的照片"→ 直接召回。
8. **一键相册**:按人物/事件/时间自动生成相册(本地 HTML 画廊或导出)。
9. **隐私优先**:人脸模型与向量**只在本地**,默认不上传;上云任何一步都需显式同意。

### 2.2 非目标
- 不做云端人脸身份库、不做跨用户/陌生人比对、不做"猜你是谁"的社交识别。
- 不做社交分享平台/在线相册托管(只产出本地相册,分享由用户自行决定)。
- 不做专业剪辑可用片段/时间码(那是 v1 的事;本版可复用但不强调)。
- 不做实时处理;以批量离线整理为主。

## 3. 目标用户与典型场景

| 用户 | 场景 | 期望 |
|------|------|------|
| 家长 | 几万张孩子成长照混在一起 | 按"孩子+年份"一键找齐、做成长相册 |
| 旅行爱好者 | 多次旅行照片混杂 | 按地点/行程自动归成相册 |
| 普通用户 | 找某次聚会/节日的照片视频 | 大白话搜:"去年中秋全家福" |
| 念旧的人 | 想回顾和某位家人的所有合影 | 按人物聚合,自动生成"和奶奶的回忆" |

**核心旅程:**
1. 指向照片来源(本地文件夹 / 云盘挂载目录 / 导出的 iCloud·Google·百度网盘目录)。
2. Agent 自动盘点、看懂、检测人脸。
3. **用户给人脸命名**(一次性,之后自动认人)。
4. 自动归类(时间/地点/场景/事件)、去重选优、命名。
5. 之后随时:大白话检索 / 浏览事件相册 / 一键导出某人/某次的相册。

## 4. 整体架构(复用 v1 管线 + 消费级扩展)

```
0 环境探测
1 盘点去重(指纹) ─┬─ 1b 地理富化(GPS→地名)
                  └─ 2 抽取(照片直读 / 视频抽帧+缩略图;视频抽脸用帧)
2b 人脸:检测→向量→聚类→【用户命名】→回填+增量认人      ← 本版核心
2c 近似去重:感知哈希分组 → 选最佳
3 理解(生活化 vocab:场景/内容/活动)
3b 事件聚类:时间+地点+人物+内容 → 事件/相册
4 打标签 + 简短命名(时间_人物_地点/事件)
5 入库(本地 HTML 画廊【主】 / 飞书 / 旁车+Excel;可选按 年/事件 归档)
6 检索 + 一键相册(自然语言 → 结构化过滤;按人/事件/时间成册)
```

> 与 v1 的关系:`0/1/2/3/4/5` 复用同一套脚本与记录契约;新增 `1b 地理`、`2b 人脸`、`2c 去重`、`3b 事件`、消费级 `5 画廊` 与 `6 相册`。统一 schema 增量扩字段,不破坏 v1。

### 4.1 人脸归类(本版核心,人在环节)

> 用**本地人脸模型**(InsightFace/ArcFace 系)做检测与 512 维向量,**不用多模态 LLM**;准确、免费、可离线、保护隐私。

1. **检测 + 向量**:扫所有照片与视频关键帧,检测人脸框 + 算向量。
2. **自动聚类**:对向量聚类(HDBSCAN / 阈值法),同一个人聚成一组,每组挑代表脸。
3. **用户命名**:呈现"人脸命名清单"(代表脸缩略图),用户为每组起名("妈妈""乐乐""奶奶")或标记"忽略/路人";支持**合并多组**(同人被拆开)与**拆分一组**(误并)。
4. **回填**:人物名回填到所有相关素材的 `persons` 字段,并参与命名与检索。
5. **增量认人**:新素材的人脸按向量相似度自动匹配已命名的人;匹配不上进"待命名"。
6. **隐私**:人脸向量与人名映射只存本地 `state/faces.db`,不外传、不对外比对。

### 4.2 地点(GPS → 地名)

- 读 EXIF/视频 GPS → **反向地理编码**得国家/城市/地标。
- **隐私两档**:① 离线地名库(完全本地,精度到城市/区);② 在线地理编码 API(更准到地标,需用户同意,因为会把坐标发出去)。默认离线。

### 4.3 事件聚类(自动成册的基础)

- 输入:每张素材的 时间 + 地点 + 人物 + 内容。
- 规则 + 模型结合:**时间相近 + 地点相近**先聚成候选事件,再用内容/人物给事件**起名**(如"2024-02 春节·奶奶家"、"三亚旅行 6 天")。
- 事件即相册的天然单位。

### 4.4 近似去重 + 选最佳

- **感知哈希(pHash/dHash)** 把视觉高度相似的照片(连拍/重复)分到一组。
- 组内**选最佳**:综合画质(清晰度/曝光)、人脸是否清晰、是否闭眼/表情,标 `is_best`;其余标 `similar_group` 但保留(不删,用户可一键清理)。

### 4.5 数据层与消费入口

- **本地 HTML 画廊(主推)**:生成一个可在浏览器打开的相册站点——按 人物 / 事件 / 时间 / 地点 浏览与筛选,缩略图预览。**零依赖、完全本地、隐私友好**,对普通人最友好。
- 仍支持 **飞书多维表格 / JSON 旁车 + Excel**(复用 v1 适配器)。
- 可选**物理归档**:按 `年份/事件` 复制或软链到整理后的目录(默认不动原文件)。

## 5. 数据 Schema(在 v1 基础上扩展)

> 复用 v1 `schema/record.schema.json`,新增消费级字段(对 v1 可选,不破坏)。

| 新增字段 | 类型 | 说明 |
|------|------|------|
| `persons` | string[] | 已命名人物(来自人脸归类),如 ["妈妈","乐乐"] |
| `face_ids` | string[] | 关联的人脸聚类 id(内部用) |
| `place` | object | `{name, city, country, gps}` 反查地名 |
| `event_id` | string | 所属事件 |
| `event_name` | string | 事件名(如"2024春节·奶奶家") |
| `burst_group` | string | 近似照片分组 id |
| `is_best` | bool | 该近似组里是否最佳 |
| `taken_date` | date | 拍摄日期(检索/事件用,源自 shot_at) |

**人物表(本地 faces.db / 可同步到飞书"人物表"):**

| 字段 | 说明 |
|------|------|
| `person_id` | 聚类 id |
| `name` | 用户起的名字(空=待命名) |
| `representative_face` | 代表脸缩略图 |
| `face_count` | 关联人脸数 |
| `relation` | 可选:关系标签(家人/朋友…) |
| `status` | named / pending / ignored |

### 5.1 生活化受控词表(vocab.consumer)
- `scene`:家 / 餐厅 / 公园 / 海边 / 山区 / 街道 / 景点 / 学校 / 办公室 / 车内 / 节日现场 …
- `occasion`(活动/场合):生日 / 婚礼 / 旅行 / 聚餐 / 节日 / 毕业 / 运动 / 演出 / 日常 …
- 复用 v1 的 quality 概念(选最佳用)。

### 5.2 命名规则(简短)
- 照片:`{date}_{persons}_{place或event}_{seq}` → `20240715_乐乐_三亚海边_01.jpg`
- 视频同理加可选标识。
- 缺字段自动省略;非法字符过滤;可回滚(复用 v1 安全改名引擎)。

## 6. 检索与一键相册

### 6.1 自然语言检索
用户输入大白话 → 文本模型解析成结构化过滤条件(人物 / 时间范围 / 地点 / 场景 / 事件 / 关键词)→ 在库上结构化过滤 + 语义排序 → 返回结果(画廊/列表)。

例:"去年夏天乐乐在海边的照片" → `persons=乐乐, date∈2024-06~08, scene=海边` → 召回。

### 6.2 一键相册
- 按 **人物**:"乐乐的成长相册"(按时间排序,挑 is_best)。
- 按 **事件**:"三亚旅行"相册。
- 按 **时间**:"2024 年度精选"。
- 输出:本地 HTML 画廊页 / 导出图片集 / (可选)生成简单视频集锦脚本喂给 v1。

---

## 7. 隐私(消费级红线)

- **人脸**:检测/向量/聚类/人名映射**全本地**,默认绝不上传;无云端身份库、无跨库比对。
- **地理**:默认离线地名库;在线地理编码需显式同意(因坐标会外发)。
- **内容理解**:照片送多模态模型属"上云",**必须显式告知并同意**,并提供"仅本地模型"选项(用本地多模态替代,牺牲部分效果)。
- **数据**:所有索引/缩略图/相册默认存本地;删除原文件需多重确认。

## 9. 风险与对策

| 风险 | 对策 |
|------|------|
| 人脸聚类不准(同人多组/异人并组) | 阈值可调 + 命名时支持合并/拆分 + 代表脸人工确认 + 增量复用 |
| 规模大(几万张)性能 | 向量索引(faiss/annoy)+ 分批 + 缩略图加速浏览 + 断点续跑 |
| HEIC/RAW/实况照片 等格式 | 解码兼容层(pillow-heif 等);实况照片当照片+短视频处理 |
| 离线地名库精度有限 | 提供在线编码可选项(显式同意);精度档位可配 |
| 隐私顾虑(家人照片) | 本地优先 + 上云显式同意 + 本地多模态可选 + 不删原文件 |
| 视频里抽脸成本 | 视频按关键帧抽脸,限频;人脸只对清晰帧做 |

## 10. 验收标准(消费版)

1. 给定 ≥2000 张混合照片/视频,完成 看懂→人脸聚类→地理→事件→去重→命名→入库 全流程,断点续跑。
2. 人脸:能聚类、用户能命名/合并/拆分,命名后按人物准确检索;新照片能自动认人。
3. 自然语言检索"某人+某时间+某地/场景"能召回合理结果。
4. 能按 人物/事件/时间 一键生成本地 HTML 相册。
5. 默认配置下人脸数据与地理坐标**不出本地**;任何上云步骤有明确同意。
6. 近似连拍能分组并选出最佳。

## 11. 待确认 / 后续决策

- **主消费入口形态**:本地 HTML 画廊(推荐)vs 飞书表 vs 仅文件夹归档——先做哪个?
- **照片来源**:优先支持哪种?(本地文件夹 / iCloud 导出 / Google Takeout / 百度网盘挂载)
- **本地人脸模型**与运行算力(有无 GPU,影响几万张的速度)。
- **地理编码**:离线库精度档位 vs 在线 API,默认策略。
- **上云内容理解 vs 纯本地多模态**:是否要提供"完全离线"模式(牺牲效果换隐私)。
- 是否需要"实况照片(Live Photo)/连拍"的专门处理策略细化。
