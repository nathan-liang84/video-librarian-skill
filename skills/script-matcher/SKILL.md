# 脚本选素材 Skill(script-matcher)

> VideoLibrarian 家族的**第二个独立入口**:建库交给 `video-librarian`(01-05),
> 选素材就用这个。两者**同仓库、共享同一套库与模型层**,可分别独立触发。

把一段剪辑脚本 / 分镜 / 拍摄需求,变成一张"镜头 → 候选素材"的对照表:每个镜头给出
推荐文件、推荐时间码、贴合理由与评分,直接喂给剪辑。

## 何时使用

- 用户**已经用 VideoLibrarian 建过素材库**(有旁车 `.json` 或飞书表),现在拿着脚本要选素材
- 用户给一段分镜/口播稿/拍摄清单,问"哪些素材能用 / 这个镜头用哪条"
- 不负责建库:若库还没建,先走 `video-librarian` 的 01-05

## 前置条件

1. 素材库已建好(其一即可):
   - 素材旁有同名 `.json` 旁车(`store.mode: sidecar/both` 跑过 05),**或**
   - `state/manifest.json` 仍在(刚建完库的工作状态)
2. `config/config.yaml` 里 `models.text` 可用(脚本解析 + 候选排序都用文本模型)
3. 一个纯文本脚本文件(`.txt`/`.md`,一行一个镜头或自然段都行)

## 用法

```bash
# 最简:用当前工作状态(manifest)里的库
python scripts/06_match.py --script 脚本.txt

# 独立选素材:库建好后 manifest 可能已清,直接指向素材目录读旁车持久库
python scripts/06_match.py --script 脚本.txt --input /path/to/素材目录 --top 3

# 自定义报告输出位置
python scripts/06_match.py --script 脚本.txt --input /path/to/素材 --out 我的匹配.md
```

参数:
- `--script`(必填):剪辑脚本/分镜文本文件
- `--input`(可选,可重复):素材根目录,从旁车 `.json` 读**持久库**(优先于 manifest)
- `--top`:每个镜头返回候选数(默认 3)
- `--out`:匹配报告路径(默认 `output/_匹配报告.md`)

## 工作原理

```
脚本 →[文本模型 解析]→ 镜头需求(场景/人物/景别/情绪/时长)
     →[硬过滤 渐进放宽]→ 候选素材
     →[文本模型 语义排序]→ 推荐文件 + 时间码 + 理由 + 评分 → 屏幕 + Markdown 报告
```

- **读库优先级**:持久库(旁车 `.json`)> manifest 兜底。素材搬家后旁车跟着走,
  库依然在;manifest 是临时工作状态,可丢。
- **硬过滤 = 可解释**:`scene`(场景)与 `subjects`(人物)是硬约束,
  `shot_type`(景别)是软偏好。避免"模型一句话定生死"。
- **渐进放宽,绝不空手**:精确档召回为 0 时,先放景别、再放人物(场景始终锚定),
  并在结果里**透明标注**`(已放宽:景别/人物)`,让你知道这是降级匹配。
- **库里真没有就如实说**:放宽到底仍无匹配 → 标"无匹配素材",不瞎凑(不会把海边
  需求硬塞一条室内素材)。

## 输出

- 屏幕:每个镜头的候选清单(文件名 + 时间码 + 理由 + ★评分)
- 文件:`output/_匹配报告.md`(或 `--out` 指定),Markdown 表格,可直接发给剪辑/客户

## 与建库技能的关系(未来合二为一)

二者已在**同一仓库**,共享 `lib/`(模型层)、`schema/record.schema.json`、`config/`、
旁车/飞书数据层 —— 代码层面早已"合一",只是对外暴露成两个入口。
未来若要做成一个端到端命令(建库即可选素材),无需合并代码,加一层编排即可。
